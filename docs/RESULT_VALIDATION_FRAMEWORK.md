# GPT-5.6 Model Router 결과 검증 체계

## 문서 목적

이 문서는 `gpt-5.6-router`가 Sol 단독 실행과 비교해 실제로 다음 효과를 내는지 검증하기 위한 공통 기준을 정의합니다.

- 총 토큰 사용량을 줄이는가?
- 결과 품질을 같은 수준으로 유지하는가?
- 품질이 낮아진다면 그 차이가 어느 정도인가?
- 작업 시간이 단축되는가, 오히려 길어지는가?
- 재작업, 검수, 통합 비용까지 포함해도 효율적인가?
- 어떤 작업에서 라우터가 유리하고 어떤 작업에서는 Sol Direct가 유리한가?
- PERFORMANCE, BALANCED, TOKEN_SAVER가 이름에 맞는 결과를 내는가?

이 검증 체계의 목적은 프로젝트를 홍보하기 위한 유리한 숫자를 만드는 것이 아닙니다. 라우터가 효과적인 범위와 효과적이지 않은 범위를 함께 공개할 수 있는 재현 가능한 증거를 만드는 것입니다.

적응형 오케스트레이션 개발은 이 검증 체계에서 생성된 결과만을 근거로 진행합니다. 관련 개발 방향은 [ADAPTIVE_ORCHESTRATION_DIRECTION.md](ADAPTIVE_ORCHESTRATION_DIRECTION.md)를 따릅니다.

# 검증의 핵심 질문

## 1. 토큰 절감

라우팅 경로가 Sol Direct보다 총 토큰을 얼마나 적게 사용하는지 확인합니다.

여기서 총 토큰은 하위 모델이 사용한 토큰만 의미하지 않습니다. 다음을 모두 포함해야 합니다.

- 부모 에이전트의 계획 토큰
- 프로필 선택과 라우팅 계획 토큰
- 서브에이전트 입력 토큰
- 서브에이전트 출력 토큰
- 반복 호출과 재시도 토큰
- 검수 토큰
- 결과 통합 토큰
- 실패 후 재작업 토큰

저렴한 모델을 사용했더라도 검수와 재작업 때문에 총 토큰이 늘었다면 절감으로 보지 않습니다.

## 2. 품질 유지

토큰이 줄더라도 Sol Direct보다 결과 품질이 크게 낮아지면 성공이 아닙니다.

검증은 다음을 구분합니다.

- 객관적으로 자동 확인 가능한 품질
- 사람이 판단해야 하는 주관적 품질
- 안전과 권한 경계
- 장기 유지보수 품질

## 3. 작업 시간

멀티에이전트는 병렬 실행으로 빨라질 수도 있지만 계획, 전달, 통합 때문에 더 느려질 수도 있습니다.

다음 시간을 구분해 기록합니다.

- 사용자가 작업을 요청한 시점부터 완료까지의 실제 경과 시간
- 모델이 실제로 실행된 총 처리 시간
- 병렬 실행으로 겹친 시간
- 사용자 승인을 기다린 시간
- 재시도와 재작업 시간

사용자 승인 대기 시간은 시스템 처리 시간과 분리해서 보고합니다.

## 4. 안정성과 변동성

한 번 좋은 결과가 나온 것만으로는 충분하지 않습니다. 동일 작업을 반복했을 때 결과가 얼마나 안정적인지 확인합니다.

- 성공률
- 품질 점수의 분산
- 토큰 사용량의 분산
- 작업 시간의 분산
- 모델 고정 실패율
- 라우팅 계획과 실제 실행 경로 불일치율

## 5. 적용 범위

최종 목표는 모든 작업에서 라우터가 Sol Direct보다 낫다고 주장하는 것이 아닙니다.

다음과 같은 적용 지도를 만드는 것이 목표입니다.

```text
라우터 권장
- 독립적으로 분리 가능한 중대형 작업
- 반복 변환량이 큰 작업
- 객관적 검증기가 있는 구현
- 조사와 구현을 분리할 수 있는 작업

조건부 권장
- 디자인이 포함된 일반 프로그램 제작
- 문맥이 크지만 단계별 전달이 가능한 작업

Sol Direct 권장
- 매우 작은 수정
- 단계가 강하게 결합된 작업
- 전체 문맥을 계속 유지해야 하는 작업
- 검증이 어렵고 실패 영향이 큰 작업
```

이 지도는 가설이며 실제 검증 결과에 따라 수정합니다.

# 비교 대상

모든 주요 벤치마크는 최소한 다음 네 실행 방식을 비교합니다.

## A. Sol Direct

- 전체 작업을 Sol이 직접 수행합니다.
- 서브에이전트는 사용하지 않습니다.
- 라우터의 기본 기준선입니다.

## B. PERFORMANCE

- 품질과 독립 검증을 우선합니다.
- 필요한 경우 추가 에이전트와 Sol 검토를 허용합니다.

## C. BALANCED

- 필수 품질을 유지하면서 중복 호출과 컨텍스트를 줄입니다.
- 기본 추천 프로필입니다.

## D. TOKEN_SAVER

- 안전과 품질 하한을 유지하면서 총 토큰을 최소화합니다.
- Sol 사용량만이 아니라 재작업을 포함한 총 토큰으로 평가합니다.

필요한 경우 다음 보조 기준선을 추가할 수 있습니다.

- Terra Direct
- Luna Direct
- Codex 기본 자동 서브에이전트 동작
- 라우터 이전 버전
- 새 적응형 추천 경로

보조 기준선은 반드시 주 비교인 Sol Direct를 대체하지 않습니다.

# 검증 단계

## 단계 1: 소규모 파일럿

목적:

- 측정 도구와 평가 절차가 제대로 작동하는지 확인합니다.
- 대규모 실험 전에 평가 기준의 모호성을 제거합니다.

권장 구성:

- 12개 이상의 작업
- 작업별 각 경로 최소 2회 반복
- 자동 평가와 사람 평가를 모두 포함

파일럿에서 확인할 사항:

- 토큰이 실제로 관찰되는가?
- 실행 시간이 일관되게 기록되는가?
- 동일 시작 상태를 복원할 수 있는가?
- 평가자가 결과 차이를 구분할 수 있는가?
- 작업 난이도가 지나치게 쉽거나 어렵지 않은가?
- 실패 유형이 기록 체계에 포함되는가?

파일럿 결과는 일반 성능 주장에 사용하지 않습니다.

## 단계 2: 기준 벤치마크 v1

권장 구성:

- 최소 40개 작업
- 최소 8개 작업 범주
- 범주당 5개 이상의 작업
- 각 작업과 경로를 최소 3회 반복
- 실행 순서를 무작위화

권장 작업 범주:

1. 소형 코드 수정
2. 일반 기능 구현
3. 중대형 기능 구현
4. 아키텍처 및 설계
5. 프론트엔드와 디자인
6. 테스트와 회귀 검증
7. 대량의 결정적 변환
8. 코드베이스 조사와 분석

추가 가능 범주:

- 성능 최적화
- 보안 검토
- 리팩터링
- 문서화
- 데이터 마이그레이션 계획
- 복합 도구 사용

## 단계 3: 실제 프로젝트 검증

합성된 작은 과제만으로는 실제 개발 효과를 판단하기 어렵습니다.

실제 프로젝트에서 다음을 검증합니다.

- 기존 코드베이스 이해
- 여러 파일에 걸친 변경
- 실제 테스트와 빌드
- 회귀 오류
- 사용자 요구사항 수정
- 장시간 작업에서의 컨텍스트 관리
- 실패 후 복구

실제 프로젝트 검증은 작업 원문이나 소스코드를 공개하지 않고도 구조화된 결과만 공개할 수 있어야 합니다.

## 단계 4: 지속 검증

모델과 Codex가 업데이트되면 기존 결과가 더 이상 유효하지 않을 수 있습니다.

다음 변화가 있을 때 핵심 벤치마크를 다시 실행합니다.

- Sol, Terra, Luna 모델 버전 변경
- 추론 강도 동작 변경
- Codex CLI 주요 업데이트
- Codex Desktop 런타임 변경
- Multi-Agent V2 도구 스키마 변경
- 라우팅 정책의 주요 수정
- 컨텍스트 전달 방식 변경

# 공정한 비교 조건

## 동일 시작 상태

각 경로는 동일한 저장소 커밋 또는 동일한 파일 스냅샷에서 시작해야 합니다.

- 이전 실행의 변경 사항을 다음 실행에 남기지 않습니다.
- 빌드 산출물과 캐시의 영향을 통제합니다.
- 테스트 데이터와 외부 서비스 상태를 가능한 한 고정합니다.
- 실행마다 새 브랜치, 복제본 또는 격리된 작업 디렉터리를 사용합니다.

## 동일한 작업 지시

- 사용자 요구사항은 경로마다 동일해야 합니다.
- 특정 경로에만 추가 힌트를 주지 않습니다.
- 라우터 실행에 필요한 프로필 선택 외에는 요구사항을 바꾸지 않습니다.
- 평가용 숨은 테스트는 실행 모델에 공개하지 않습니다.

## 동일한 권한과 도구

- 파일 접근 범위
- 네트워크 접근
- MCP 도구
- 테스트 명령
- 샌드박스 정책
- 외부 작업 승인

위 조건을 경로마다 동일하게 유지합니다.

## 버전 기록

각 실행에 다음을 기록합니다.

- 날짜와 시간
- 부모 모델
- 서브에이전트 모델
- 추론 강도
- Codex CLI 버전
- Codex Desktop 버전
- 운영체제
- 라우터 정책 버전
- 저장소 시작 커밋
- 테스트 도구 버전

## 실행 순서 무작위화

항상 Sol Direct부터 실행하면 시간대, 캐시, 서비스 상태의 영향을 받을 수 있습니다.

작업별 실행 순서를 무작위로 섞고 순서 자체도 기록합니다.

## 반복 실행

LLM 결과에는 변동성이 있습니다. 한 번의 결과로 결론 내리지 않습니다.

- 파일럿: 경로별 최소 2회
- 정식 벤치마크: 경로별 최소 3회
- 결과 분산이 크면 5회 이상

# 품질 평가 체계

## 1. 자동 평가

자동 평가가 가능한 작업에서는 자동 결과를 가장 먼저 사용합니다.

### 코드 작업

- 테스트 통과율
- 숨은 테스트 통과율
- 빌드 성공 여부
- 타입 검사
- 정적 분석
- 린트
- 코드 커버리지 변화
- 성능 기준 충족 여부
- 보안 검사 결과

### 프론트엔드 작업

- 지정 뷰포트 렌더링 성공
- 요소 겹침 여부
- 콘솔 오류
- 접근성 검사
- 색상 대비
- 반응형 상태
- 상호작용 테스트
- 스크린샷 차이

### 변환 작업

- 변환 성공 건수
- 누락 건수
- 중복 건수
- 형식 검증
- 체크섬 또는 구조 비교
- 예외 처리 정확도

### 조사 작업

- 알려진 정답 항목 회수율
- 근거 파일과 줄 위치 정확도
- 허위 주장 수
- 중요 항목 누락 수

## 2. 사람 평가

자동 테스트만으로 완성도와 설계 품질을 모두 평가할 수 없습니다.

사람 평가는 가능한 한 결과를 만든 경로를 모르는 상태에서 진행합니다.

평가 항목 예시:

- 요구사항 충족도
- 정확성
- 코드 가독성
- 유지보수성
- 구조 적절성
- 오류 처리
- 사용자 경험
- 시각적 완성도
- 독창성
- 설명의 명확성

권장 점수:

- 항목별 1점에서 5점
- 최종 점수는 100점 기준으로 환산 가능

## 3. 쌍대 비교

주관적 결과는 절대 점수보다 두 결과를 직접 비교하는 방식이 더 안정적일 수 있습니다.

평가자에게 다음 중 하나를 선택하게 합니다.

- A가 명확히 우수
- A가 약간 우수
- 차이 없음
- B가 약간 우수
- B가 명확히 우수

결과의 실행 경로와 모델 이름은 평가자에게 숨깁니다.

## 4. 안전 품질

다음 항목은 일반 품질 점수와 별도로 통과해야 합니다.

- 승인 범위 위반 없음
- 비밀정보 노출 없음
- 권한 확대 없음
- 승인되지 않은 외부 작업 없음
- 잘못된 모델 대체 없음
- 파괴적 작업의 별도 승인 유지

치명적 안전 실패가 한 번이라도 발생하면 해당 경로는 효율이 좋아도 출시 후보에서 제외합니다.

# 핵심 지표

## 총 토큰

```text
total_tokens = parent_input
             + parent_output
             + child_inputs
             + child_outputs
             + retry_tokens
             + review_tokens
             + integration_tokens
             + rework_tokens
```

런타임이 일부 항목만 제공한다면 관찰 범위를 명시합니다.

## 토큰 절감률

```text
token_saving_rate =
    (sol_direct_tokens - routed_tokens)
    / sol_direct_tokens
    * 100
```

해석 예시:

- 양수: 라우터가 더 적은 토큰 사용
- 0에 가까움: 유의미한 차이 없음
- 음수: 라우터가 더 많은 토큰 사용

평균만 보고하지 않고 중앙값, 사분위 범위, 반복 실행 분산을 함께 보고합니다.

## 품질 유지율

```text
quality_retention =
    routed_quality_score
    / sol_direct_quality_score
    * 100
```

Sol Direct 점수가 0이거나 매우 낮은 작업에는 이 비율을 사용하지 않고 절대 점수 차이를 사용합니다.

## 품질 차이

```text
quality_delta = routed_quality_score - sol_direct_quality_score
```

초기 분류 기준은 검증 전에 고정합니다.

- 동등 범위: -5점 이상
- 소폭 저하: -10점 이상 -5점 미만
- 유의미한 저하: -10점 미만
- 향상: 5점 초과

이 값은 초기 운영 기준이며 파일럿에서 평가 척도의 분산을 확인한 뒤 정식 벤치마크 전에 확정합니다. 결과를 본 뒤 유리하게 변경하지 않습니다.

## 작업 성공률

```text
success_rate = successful_runs / valid_runs * 100
```

환경 장애와 측정 실패는 별도 분류하고 무조건 실패나 성공으로 포함하지 않습니다.

## 시간 절감률

```text
time_saving_rate =
    (sol_direct_elapsed - routed_elapsed)
    / sol_direct_elapsed
    * 100
```

사용자 승인 대기 시간을 포함한 값과 제외한 값을 함께 보고합니다.

## 재작업률

```text
rework_rate = runs_requiring_material_rework / completed_runs * 100
```

## 오케스트레이션 오버헤드

```text
orchestration_overhead =
    routing_planning_tokens
    + handoff_tokens
    + integration_tokens
    + routing_validation_tokens
```

라우터가 절감한 실행 토큰보다 오케스트레이션 오버헤드가 크면 비효율 경로로 분류할 수 있습니다.

## 모델 고정 신뢰도

```text
pinning_success_rate =
    runs_with_verified_requested_model
    / runs_requesting_model_override
    * 100
```

모델 정보를 관찰할 수 없는 실행은 성공으로 간주하지 않습니다.

# 결과 판정 기준

## 성공적인 라우팅

다음 조건을 모두 만족할 때 성공적인 효율 개선으로 봅니다.

- 필수 테스트와 안전 검증 통과
- 품질이 사전에 정한 동등 또는 비열등 범위 안에 있음
- 총 토큰 또는 실제 경과 시간에서 반복적인 이익이 있음
- 재작업률이 허용 범위를 넘지 않음
- 모델 고정과 실제 경로가 검증됨
- 특정 한 작업이 아니라 여러 작업에서 같은 경향이 관찰됨

## 품질은 같지만 절감 없음

- 라우터가 안전하게 작동했지만 효율 개선은 증명되지 않은 상태입니다.
- 기능적 성공과 최적화 성공을 구분해 보고합니다.

## 토큰은 줄었지만 품질 저하

품질 저하 수준에 따라 다음처럼 분류합니다.

- 허용 범위 내 저하: 프로필 목적에 따라 조건부 허용
- 유의미한 저하: 해당 작업 유형에서는 권장하지 않음
- 치명적 실패: 해당 경로 비활성화

TOKEN_SAVER라도 필수 안전과 품질 하한 아래로 내려갈 수 없습니다.

## 품질은 향상됐지만 토큰 증가

- PERFORMANCE에서는 유효할 수 있습니다.
- BALANCED에서는 품질 이익이 추가 비용을 정당화하는지 별도 판단합니다.
- TOKEN_SAVER에서는 기본적으로 실패로 봅니다.

## 라우터가 더 나쁜 작업

라우터가 불리한 작업도 반드시 공개합니다.

예시:

- 매우 작은 수정
- 전체 문맥이 계속 필요한 작업
- 인계 과정에서 정보 손실이 큰 작업
- 검수 비용이 구현 비용보다 큰 작업
- 서브에이전트 결과 통합이 어려운 작업

# 프로필별 검증 가설

## PERFORMANCE 가설

- Sol Direct와 비슷하거나 더 높은 품질
- 더 많은 토큰 또는 시간이 들 수 있음
- 고위험 또는 복잡한 작업에서 실패율 감소

검증 실패 사례:

- 토큰만 늘고 품질과 성공률 차이가 없음
- 추가 검토가 중복되어 결과를 개선하지 못함

## BALANCED 가설

- Sol Direct 대비 품질 점수 차이가 동등 범위 안
- 총 토큰 감소
- 작업 시간은 비슷하거나 감소
- 재작업률 증가가 작음

핵심 검증 대상은 BALANCED입니다.

## TOKEN_SAVER 가설

- 필수 품질과 안전 하한 유지
- Sol Direct와 BALANCED보다 총 토큰 감소
- 결정적이고 반복적인 작업에서 가장 큰 이익

검증 실패 사례:

- Luna 또는 Terra 결과를 Sol이 다시 고치면서 총 토큰 증가
- 단순한 모델 비용만 낮고 재작업률이 크게 증가

- 규모
- 모호성
- 실패 영향
- 문맥 크기
- 단계 결합도

- 디자인 판단 필요성
- 반복량
- 외부 도구 의존성

작업이 끝난 뒤 결과를 보고 난이도 분류를 바꾸지 않습니다. 잘못 분류된 경우 원래 분류와 사후 평가를 모두 기록합니다.

- 통합 실패

## 평가 실패

- 토큰 일부 누락
- 환경 장애
- 동일 시작 상태 복원 실패
- 숨은 테스트 오염

# 통계 보고 원칙

## 중앙값 중심 보고

토큰과 시간은 이상치의 영향을 많이 받으므로 평균뿐 아니라 중앙값을 중심으로 보고합니다.

필수 보고 항목:

- 표본 수
- 중앙값
- 평균
- 최솟값과 최댓값
- 사분위 범위
- 표준편차 또는 강건한 변동성 지표

## 짝지은 비교

같은 작업의 Sol Direct 결과와 라우터 결과를 짝으로 비교합니다.

서로 다른 작업의 평균만 비교하면 작업 난이도 차이가 결과를 왜곡할 수 있습니다.

## 신뢰 구간

가능하면 부트스트랩 방식으로 다음 차이의 신뢰 구간을 계산합니다.

- 품질 점수 차이
- 토큰 절감률
- 시간 절감률
- 성공률 차이

## 효과 크기

통계적 유의성만이 아니라 실제 차이의 크기를 보고합니다.

- 토큰이 1% 줄어든 결과는 표본이 많아도 실용적 의미가 작을 수 있습니다.
- 품질이 0.2점 높아졌지만 토큰이 두 배라면 BALANCED의 성공이라고 보기 어렵습니다.

## 결과를 본 뒤 기준 변경 금지

다음 항목은 정식 실행 전에 확정합니다.

- 품질 동등 범위
- 최소 절감률
- 허용 재작업률
- 제외 기준
- 필요한 표본 수
- 평가 가중치

결과를 본 뒤 유리하게 기준을 바꾸면 변경 이유와 이전 기준 결과를 함께 공개합니다.

라우터의 효율을 공개적으로 주장하려면 최소한 다음을 제공해야 합니다.

- 작업 범주와 개수
- 경로별 반복 횟수
- 모델과 Codex 버전


# 적응형 오케스트레이션과의 연결

검증 결과는 적응형 정책의 입력으로 사용할 수 있지만, 다음 단계를 지킵니다.

1. 검증 데이터 수집
2. 오프라인 경로 점수 계산
3. 기존 고정 정책과 비교
4. 그림자 추천
5. 충분한 표본과 품질 확인
6. 사용자 승인 아래 제한적 추천
7. 실패 시 기존 정책으로 롤백

검증 결과가 부족한 상태에서 자동 라우팅 가중치를 바꾸지 않습니다.

# 구현 우선순위

## P0: 측정 가능성 확인

- 중대형 실제 개발 작업

- 프로필별 품질과 효율 비교
- 작업 범주별 권장 경로
- 버전별 성능 변화
- 실패율과 신뢰도

---

# 완료 정의

결과 검증 체계가 구축되었다고 판단하려면 다음 조건을 만족해야 합니다.

1. Sol Direct와 세 프로필을 동일 조건에서 반복 실행할 수 있습니다.
2. 총 토큰에 계획, 인계, 검수, 통합, 재작업 비용이 포함됩니다.
3. 품질을 자동 평가와 블라인드 사람 평가로 측정할 수 있습니다.
4. 작업 시간과 사용자 승인 대기 시간을 분리할 수 있습니다.
5. 결과의 평균뿐 아니라 중앙값과 변동성을 확인할 수 있습니다.
6. 모델 고정 실패와 라우팅 불일치를 별도로 기록합니다.
7. 라우터가 더 나빴던 작업도 공개할 수 있습니다.
8. 모델과 Codex 버전이 바뀌면 핵심 검증을 다시 실행할 수 있습니다.
9. 공개 주장이 실제 측정 결과의 범위를 넘지 않습니다.
10. 검증 데이터가 적응형 오케스트레이션의 안전한 근거로 사용됩니다.

---

## 핵심 결론

앞으로 `gpt-5.6-router`의 가장 중요한 작업은 라우팅 규칙을 더 복잡하게 만드는 것이 아닙니다.

먼저 Sol Direct와 같은 조건에서 비교해 다음을 사실로 확인해야 합니다.

- 어디에서 토큰을 절약하는가?
- 절감률은 어느 정도인가?
- 품질은 같은가, 조금 낮은가, 크게 낮은가?
- 검수와 재작업을 포함해도 이익인가?
- 어떤 작업에서는 Sol Direct를 유지해야 하는가?

이 결과가 축적된 뒤에만 프로젝트는 단순한 규칙 기반 스킬에서 측정 가능하고 적응 가능한 라우팅 시스템으로 발전할 수 있습니다.
