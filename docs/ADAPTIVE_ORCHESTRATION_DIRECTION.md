# 적응형 오케스트레이션 개발 방향

## 문서 목적

이 문서는 `gpt-5.6-router`를 현재의 **사람이 설계한 고정 라우팅 정책**에서, 실제 실행 결과를 관찰하고 점진적으로 정책을 보정하는 **증거 기반 적응형 오케스트레이션 시스템**으로 발전시키기 위한 중장기 개발 방향을 정의합니다.

현재 라우터의 규칙 기반 설계는 안전한 시작점입니다. Sol, Terra, Luna의 역할 경계와 사용자 승인, 모델 고정 검증, 실패 폐쇄 원칙을 명확히 정의하기 때문입니다. 그러나 모델 성능, Codex 런타임, 작업 유형, 컨텍스트 크기, 검증 가능성은 계속 달라질 수 있습니다. 고정 규칙만으로는 이러한 변화에 지속적으로 최적화하기 어렵습니다.

향후 목표는 고정 정책을 즉시 제거하는 것이 아닙니다. 먼저 실제 결과를 충분히 측정하고, 학습된 추천을 기존 정책 옆에서 비교한 뒤, 검증된 범위에서만 제한적으로 적응하도록 발전시킵니다.

---

## 연구 배경

2026년 5월 공개된 **Learning to Orchestrate Agents under Uncertainty** 연구는 BOT-Orch라는 적응형 오케스트레이션 방식을 제안합니다. 이 연구는 서로 다른 에이전트의 다음 특성을 함께 고려해 순차적으로 위임 대상을 조정합니다.

- 과거 작업 성공률과 신뢰도
- 실행 비용
- 출력 결과의 불확실성
- 작업과 에이전트 출력 분포의 정렬 정도
- 시간에 따라 달라지는 에이전트 행동

BOT-Orch는 오케스트레이션을 다중 선택 문제로 보고, 실행 결과가 누적될수록 에이전트 선택 정책을 업데이트합니다. 논문은 표준 밴딧 및 고정 휴리스틱보다 좋은 결과를 보고했지만, 주요 실험은 합성 또는 반합성 환경과 적대적으로 구성된 작업 배정 환경에서 이루어졌습니다.

따라서 다음을 구분해야 합니다.

1. **연구가 보여 준 방향성**  
   고정된 규칙만 사용하는 것보다 실제 신뢰도, 비용, 불확실성을 반영해 정책을 조정하는 방식이 유리할 수 있습니다.

2. **아직 증명되지 않은 부분**  
   같은 방식이 Codex 기반 실제 소프트웨어 개발에서 Sol, Terra, Luna의 품질과 토큰 효율을 자동으로 개선한다고 단정할 수 없습니다.

3. **이 프로젝트가 취할 접근**  
   논문의 수학적 방법을 그대로 복제하기보다, 실제 Codex 개발 작업에서 관찰 가능한 지표부터 수집하고 단계적으로 적응 기능을 검증합니다.

참고 연구:

- Learning to Orchestrate Agents under Uncertainty, arXiv:2605.27073
- Uno-Orchestra: Parsimonious Agent Routing via Selective Delegation, arXiv:2605.05007
- AdaptOrch: Task-Adaptive Multi-Agent Orchestration in the Era of LLM Performance Convergence, arXiv:2602.16873

## 현재 상태

현재 `gpt-5.6-router`는 다음과 같은 고정 정책을 사용합니다.

- 작업을 설계, 조사, 구현, 반복 변환, 검증, 최종 판단 단계로 분해합니다.
- 각 단계의 모호성, 실패 영향, 반복성, 검증 가능성을 기준으로 모델 하한을 정합니다.
- Sol은 설계, 아키텍처, 고위험 판단, 주관적 품질 승인을 담당합니다.
- Terra는 일반 구현, 조사, 테스트, 증거 정리를 담당합니다.
- Luna는 객관적으로 검증 가능한 반복 변환 작업을 담당합니다.
- 부모 모델을 하나의 활성 역량으로 계산해 중복 호출을 줄입니다.
- 작은 작업이나 문맥 결합도가 높은 작업에는 Sol 직접 실행을 권장합니다.
- 모델 고정이 검증되지 않으면 실행하지 않습니다.

이 정책은 안전성과 설명 가능성이 높지만 다음 한계가 있습니다.

- 실제 작업별 성공률이 정책에 자동 반영되지 않습니다.
- 모델 업데이트 후 달라진 성능을 즉시 반영하지 못합니다.
- 어떤 작업에서 Terra가 Sol과 비슷한지 경험적으로 학습하지 않습니다.
- Luna를 사용했을 때 검수 비용이 더 커지는 상황을 자동으로 기억하지 않습니다.
- 각 프로필의 품질과 토큰 절감 효과가 충분히 측정되지 않았습니다.
- 고정된 디자인 모델 하한이 일부 단순 UI 작업에서는 과도할 수 있습니다.

# 목표 상태

장기적으로 라우터는 다음 질문에 실행 이력을 근거로 답할 수 있어야 합니다.

1. 이 작업은 Sol 직접 실행이 더 나은가, 단계별 위임이 더 나은가?
2. 이 단계는 Terra로 처리해도 Sol 대비 품질 손실이 허용 범위 안인가?
3. Luna를 사용하면 실제로 토큰이 줄어드는가, 아니면 검수와 재작업 때문에 더 많이 드는가?
4. 특정 작업 유형에서 어떤 모델과 추론 강도가 가장 안정적인가?
5. 현재 추천의 근거가 충분한가, 아니면 데이터가 부족해 기존 보수적 정책을 유지해야 하는가?
6. 최근 Codex 또는 모델 업데이트 이후 기존 성능 관계가 달라졌는가?
7. 적응형 추천이 품질, 보안, 승인 경계를 침해하지 않는가?

# 핵심 개발 원칙

## 1. 안전 규칙은 학습 대상이 아닙니다

다음 규칙은 실행 데이터가 어떤 결과를 보여도 자동으로 완화하지 않습니다.

- 사용자의 명시적 실행 승인
- 외부 작업과 파괴적 작업의 별도 승인
- 부모 샌드박스와 권한 정책 유지
- 모델 고정 실패 시 중단
- 비밀정보와 소스코드의 자동 수집 금지
- 서브에이전트의 하위 에이전트 생성 금지
- 고위험 보안 판단과 파괴적 결정의 품질 하한
- 검증되지 않은 모델 대체 금지

적응형 정책은 **허용된 안전 범위 안에서 어떤 합법적인 경로가 더 나은지**만 학습합니다.

## 2. 측정이 학습보다 먼저입니다

정확한 결과 측정 없이 라우팅 정책부터 자동으로 바꾸지 않습니다.

우선 다음 순서를 따릅니다.

1. 실행 결과를 일관된 형식으로 기록합니다.
2. Sol 직접 실행과 라우팅 결과를 같은 조건에서 비교합니다.
3. 데이터 품질과 평가자 신뢰도를 확인합니다.
4. 오프라인에서 추천 정책을 계산합니다.
5. 실제 실행에는 영향을 주지 않는 그림자 모드로 비교합니다.
6. 충분히 검증된 범위에서만 제한적 적응을 허용합니다.

## 3. 품질 하한을 먼저 만족시킵니다

비용이 낮더라도 품질 하한을 만족하지 못하는 경로는 추천하지 않습니다.

우선순위는 다음과 같습니다.

1. 안전 및 필수 정확성
2. 작업 완료와 검증 통과
3. 허용 가능한 품질 유지
4. 그 안에서 토큰, 시간, 비용 최적화

## 4. 데이터가 부족하면 기존 정책으로 돌아갑니다

새로운 작업 유형, 새로운 모델 버전, 충분하지 않은 표본에서는 적응형 정책의 확신이 낮습니다. 이 경우 현재의 보수적인 규칙 기반 라우팅을 사용합니다.

## 5. 추천의 근거를 설명할 수 있어야 합니다

적응형 라우터는 최소한 다음을 보고해야 합니다.

- 비교한 유사 작업 수
- 각 경로의 성공률
- 품질 차이
- 토큰과 시간 차이
- 결과 변동성
- 추천 확신도
- 데이터가 오래되었는지 여부

## 작업 특성

작업 원문 전체를 저장하지 않고도 다음과 같은 구조화된 특성을 기록할 수 있습니다.

- 작업 범주: 조사, 설계, 구현, 테스트, 변환, 검토
- 작업 규모: 소형, 중형, 대형
- 문맥 크기 구간
- 변경 파일 수 구간
- 단계 간 결합도
- 모호성 수준
- 오류 발생 시 영향 수준
- 객관적 검증기 존재 여부
- 디자인 표면 존재 여부
- 새로운 디자인 판단 필요 여부
- 반복 작업량 구간
- 외부 도구 사용 여부
- 읽기 전용 또는 쓰기 작업 여부

- 선택 프로필
- 부모 모델과 추론 강도
- 각 단계의 역할
- 각 단계의 모델과 추론 강도
- 에이전트 수
- 병렬 또는 순차 실행 여부
- 전달된 컨텍스트 크기 구간
- 모델 고정 검증 결과
- 승인된 경로와 실제 경로의 차이

## 결과 품질

- 작업 완료 여부
- 테스트 통과율
- 정적 분석 결과
- 빌드 성공 여부
- 회귀 실패 수
- 재작업 횟수
- 사람 평가 점수
- 디자인 품질 평가 점수
- 치명적 오류 여부
- 잘못된 파일 수정 여부
- 승인 범위 위반 여부

## 효율

- 총 입력 토큰
- 총 출력 토큰
- 총 토큰
- 에이전트별 토큰
- 실제 경과 시간
- 모델 호출 수
- 컨텍스트 중복량 추정치
- 재시도 횟수

정확한 토큰 사용량을 런타임에서 관찰할 수 없으면 `not_observable`로 기록합니다. 사용량 비율 변화나 대략적인 화면 수치를 정확한 토큰 수로 변환하지 않습니다.

## 불확실성

- 동일 작업 반복 실행 간 결과 분산
- 평가자 간 점수 차이
- 모델 출력 간 불일치
- 유사 작업 표본 수
- 최근 모델 또는 Codex 버전 변경 여부
- 추천에 사용한 데이터의 최신성
- 검증 결과의 신뢰 구간

# 효용 함수의 기본 방향

적응형 정책은 단순히 가장 저렴한 경로를 고르지 않습니다. 개념적으로 다음 항목을 함께 고려합니다.

```text
utility = quality_reward
          - token_penalty
          - time_penalty
          - failure_risk_penalty
          - uncertainty_penalty
          - orchestration_overhead
```

각 프로필은 같은 지표를 사용하되 가중치가 다릅니다.

## PERFORMANCE

- 품질과 독립 검증에 가장 높은 가중치
- 토큰과 시간 증가는 정당한 품질 이익이 있을 때 허용
- 불확실성이 높으면 보수적으로 Sol 또는 독립 검토 선택

## BALANCED

- 필수 품질 하한을 유지하면서 총 토큰과 중복 컨텍스트 감소
- 기본 추천 프로필
- 동일 품질 범위에서 더 단순한 경로 선호

## TOKEN_SAVER

- 품질 하한과 안전 경계를 만족하는 경로 중 토큰이 가장 적은 경로 선호
- 재작업과 검수까지 포함한 총 토큰으로 평가
- 저가 모델 사용 자체를 성공으로 보지 않음

# 단계별 개발 계획

## 단계 0: 고정 정책 기준선 유지

목표:

- 현재 규칙 기반 라우터를 안정적인 기준선으로 유지합니다.
- 향후 적응형 정책이 실제로 더 나은지 비교할 수 있게 합니다.

작업:

- 정책 버전 식별자 추가
- 실행 결과에 정책 버전 기록
- 모델과 Codex 런타임 버전 기록
- Sol 직접 실행 기준선 정의
- 결과 검증 체계와 공통 데이터 형식 확정

완료 기준:

- 모든 실험 결과가 어떤 정책과 런타임에서 생성됐는지 추적할 수 있습니다.

## 단계 1: 관찰 전용 측정

목표:

- 라우팅 정책을 변경하지 않고 실제 결과만 축적합니다.

작업:

- 선택적 로컬 측정 기능 설계
- 원문 프롬프트와 소스코드를 저장하지 않는 구조화 데이터 형식 구현
- 실행 경로, 품질, 시간, 토큰, 실패 유형 기록
- 사용자가 측정 기록을 끌 수 있도록 기본 로컬 제어 제공
- 측정 불가능한 값은 명시적으로 구분

완료 기준:

- 동일 작업을 반복해도 비교 가능한 결과 레코드가 만들어집니다.
- 민감한 원문이 자동 저장되지 않습니다.

## 단계 2: 오프라인 경로 점수표

목표:

- 실행 중인 라우터를 바꾸지 않고 축적된 결과로 경로별 점수를 계산합니다.

작업:

- 작업 범주별 Sol Direct, PERFORMANCE, BALANCED, TOKEN_SAVER 결과 집계
- 모델과 추론 강도 조합별 성공률 계산
- 품질 유지율과 토큰 절감률 계산
- 표본 수와 변동성을 포함한 신뢰도 표시
- 실패율이 높은 경로 자동 제외

완료 기준:

- 특정 작업 유형에서 어떤 경로가 우수했는지 오프라인 보고서로 확인할 수 있습니다.

## 단계 3: 그림자 추천 모드

그림자 모드는 적응형 정책이 추천만 계산하고 실제 실행 경로에는 영향을 주지 않는 단계입니다.

목표:

- 기존 고정 정책과 적응형 추천을 같은 작업에서 비교합니다.

작업:

- 기존 라우터가 선택한 경로 기록
- 적응형 후보가 추천한 경로 별도 기록
- 실제 결과를 기준으로 어느 추천이 더 나았을지 사후 비교
- 추천이 기존 안전 규칙을 위반하면 즉시 무효 처리
- 새로운 작업 유형에서는 기존 정책 유지

완료 기준:

- 충분한 표본에서 적응형 추천이 기존 정책보다 품질 또는 효율을 개선하는지 확인할 수 있습니다.
- 실제 사용자 작업에는 자동 변경이 발생하지 않습니다.

## 단계 4: 제한적 적응형 추천

목표:

- 위험이 낮고 검증이 쉬운 작업에서만 적응형 경로를 사용자에게 제안합니다.

초기 허용 범위 예시:

- 객관적 테스트가 있는 일반 구현
- 대량의 결정적 변환 작업
- 읽기 전용 조사
- 실패 시 자동으로 원상복구 가능한 작업

초기 제외 범위:

- 보안 설계
- 데이터 삭제와 마이그레이션
- 배포와 외부 게시
- 새로운 제품 디자인의 최종 승인
- 법률, 의료, 금융 등 고위험 판단
- 모델 고정 여부를 검증할 수 없는 런타임

작업:

- 적응형 추천 근거와 신뢰도 표시
- 기존 경로를 항상 대안으로 제시
- 사용자가 명시적으로 승인한 경우에만 추천 경로 실행
- 품질 저하 또는 실패 시 즉시 기존 보수 정책으로 복귀

완료 기준:

- 적응형 추천이 기존 승인 체계 안에서만 작동합니다.
- 안전 경계와 품질 하한이 유지됩니다.

## 단계 5: 제한적 온라인 정책 보정

목표:

- 충분히 검증된 저위험 영역에서 최근 실행 결과를 반영해 추천 가중치를 점진적으로 조정합니다.

작업:

- 작업 유형별 최근 결과에 더 높은 가중치 적용
- 모델 업데이트 후 과거 데이터의 가중치 축소
- 탐색 비율 상한 설정
- 실패 경로의 재선택 대기 기간 설정
- 표본이 부족한 경로의 자동 선택 금지
- 사용자 승인 없이 모델, 권한, 작업 범위를 변경하지 않음

완료 기준:

- 온라인 보정이 품질 저하 없이 기준선보다 나은 효율을 반복적으로 보여 줍니다.
- 정책 변화 내역을 재현할 수 있습니다.

## 단계 6: 비정상성과 성능 변화 대응

모델과 런타임은 시간이 지나며 달라질 수 있습니다. 과거에 Terra가 잘하던 작업이 새 버전에서 달라질 수 있고, Sol과 Luna의 상대적 속도와 품질도 변할 수 있습니다.

작업:

- 모델 버전별 결과 분리
- Codex CLI와 Desktop 런타임별 결과 분리
- 최근 결과와 장기 평균의 차이 감지
- 실패율 급증 시 적응형 정책 자동 비활성화
- 호환성 변경 후 기준 벤치마크 재실행
- 오래된 결과에 대한 만료 정책 정의

완료 기준:

- 모델 또는 런타임 변화가 감지되면 과거 추천을 무조건 재사용하지 않습니다.

# 적응 단위

초기에는 전체 작업 하나를 한 모델에 보내는 방식보다 **작업 단계별 경로**를 학습 대상으로 봅니다.

예시:

```text
작업 유형: 중형 웹 애플리케이션
후보 A: Sol Direct
후보 B: Sol 설계 → Terra 구현 → Sol 승인
후보 C: Sol 설계 → Terra 구현 및 테스트 → 부모 Sol 통합
후보 D: Terra 구현 → Sol 검토
```

각 후보는 다음 조합으로 식별합니다.

- 작업 단계
- 역할
- 모델
- 추론 강도
- 순서
- 병렬성
- 검증 방식
- 전달 컨텍스트 크기

같은 모델 이름이라도 역할과 문맥 전달 방식이 다르면 별도 경로로 취급합니다.

---

# 탐색과 활용

적응형 시스템은 이미 좋은 것으로 알려진 경로를 사용하는 **활용**과, 더 나은 경로를 찾기 위한 **탐색**의 균형이 필요합니다.

하지만 실제 개발 작업에서 무제한 탐색은 위험합니다. 따라서 다음 제한을 둡니다.

- 탐색은 저위험 작업에서만 허용
- 파괴적 또는 외부 작업에는 탐색 금지
- 품질 하한을 만족한 후보끼리만 비교
- 사용자 승인 없이 탐색 경로 실행 금지
- 새 모델 또는 새 경로는 처음에 그림자 모드로 평가
- 한 번의 성공만으로 정책 변경 금지
- 실패가 발생하면 탐색 가중치 즉시 축소

---

# 콜드 스타트 전략

실행 이력이 없는 초기 상태에서는 적응형 추천을 신뢰할 수 없습니다.

초기에는 다음 순서로 동작합니다.

1. 현재 `routing-policy.md`의 고정 규칙 사용
2. 유사한 작업 범주의 검증 결과가 충분한지 확인
3. 충분하지 않으면 기존 정책 유지
4. 충분하면 오프라인 점수표를 참고해 후보 추천
5. 신뢰도가 낮으면 Sol Direct 또는 보수적 경로 선택

콜드 스타트에서 Luna나 Terra를 공격적으로 사용해 데이터를 모으지 않습니다. 사용자의 실제 프로젝트를 실험 대상으로 취급하지 않습니다.

# 실패와 보상 정의

적응형 정책은 무엇을 성공으로 볼지 명확해야 합니다.

## 성공 신호

- 요구사항 완료
- 테스트와 빌드 통과
- 승인 범위 준수
- 치명적 회귀 없음
- 사람 평가에서 기준 품질 유지
- 재작업 없이 완료
- Sol 직접 대비 토큰 또는 시간 절감

## 실패 신호

- 테스트 실패
- 요구사항 누락
- 잘못된 모델 실행
- 승인 범위 초과
- 보안 또는 권한 경계 위반
- 재작업으로 총 토큰이 기준선보다 증가
- 결과를 통합하지 못함
- 주관적 품질이 허용 기준보다 낮음

## 불확실한 결과

- 토큰 사용량을 관찰할 수 없음
- 평가자가 합의하지 못함
- 환경 문제로 실행 중단
- 외부 서비스 장애
- 기준선 실행과 조건이 달라짐

불확실한 결과는 성공이나 실패로 강제 분류하지 않고 별도로 기록합니다.

# 개인정보와 데이터 관리

적응형 기능은 기본적으로 로컬 중심으로 설계합니다.

자동 저장 금지 항목:

저장 가능한 구조화 항목 예시:

- 경로 식별자
- 모델과 추론 강도
- 토큰 수

- 테스트 통과 여부
- 재작업 횟수
- 익명화된 평가 점수
- 모델 및 런타임 버전

사용자는 측정 기록을 확인하고 삭제할 수 있어야 합니다.

# 설명 가능성 요구사항

적응형 추천은 다음 형식으로 설명할 수 있어야 합니다.

```text
추천 경로: Sol 설계 → Terra 구현 → Sol 승인
근거:
- 유사 작업 18건
- Sol Direct 대비 중앙 토큰 24% 감소
- 품질 점수 차이 -1.2점
- 테스트 통과율 동일
- 재작업률 3% 증가
- 추천 신뢰도: 중간
주의:
- 현재 모델 버전 결과는 7건으로 표본이 적음
```

수치가 관찰되지 않았거나 표본이 부족하면 이를 숨기지 않습니다.

# 적응형 정책의 출시 조건

적응형 추천 기능은 다음 조건을 모두 만족하기 전에는 기본 실행 경로에 영향을 주지 않습니다.

- 결과 검증 체계가 구현됨
- Sol Direct 기준선 데이터가 있음
- 최소 표본 수 기준이 사전에 정의됨
- 품질 동등성 또는 비열등성이 확인됨
- 토큰 또는 시간 이익이 반복적으로 관찰됨
- 치명적 실패가 없음
- 모델 고정과 실행 경로 검증이 가능함
- 개인정보 보호 검토가 완료됨
- 그림자 모드에서 기존 정책보다 나쁘지 않음
- 롤백 방법이 검증됨

# 완료 정의

이 개발 방향이 구현되었다고 판단하려면 다음이 가능해야 합니다.

1. Sol Direct와 라우팅 경로를 같은 조건에서 반복 비교할 수 있습니다.
2. 작업 유형별 품질, 토큰, 시간, 실패율을 확인할 수 있습니다.
3. 적응형 추천이 어떤 데이터에 근거했는지 설명할 수 있습니다.
4. 데이터가 부족하면 자동으로 기존 규칙 기반 정책으로 돌아갑니다.
5. 모델이나 Codex 업데이트 후 성능 변화가 감지됩니다.
6. 적응형 정책이 사용자 승인과 안전 경계를 우회하지 않습니다.
7. 실제 데이터에서 품질을 유지하면서 효율을 개선한 결과를 공개 보고서로 제시할 수 있습니다.

## 핵심 결론

`gpt-5.6-router`의 다음 단계는 규칙을 더 많이 추가하는 것이 아니라, 현재 규칙이 실제로 언제 효과적이고 언제 비효율적인지 측정하는 것입니다.

고정 정책은 안전한 기준선으로 유지합니다. 그 위에 실행 결과를 관찰하는 체계, 오프라인 평가, 그림자 추천, 제한적 적응을 순서대로 추가합니다. 적응형 오케스트레이션은 목표이지만, 충분한 실제 증거가 확보되기 전에는 자동 최적화라고 주장하지 않습니다.
