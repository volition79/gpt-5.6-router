# GPT-5.6 Model Router 개발 로드맵

이 문서는 `gpt-5.6-router`를 **정책 설계가 잘 된 Codex 스킬**에서 **측정 가능하고 재현 가능한 멀티에이전트 라우팅 프로젝트**로 발전시키기 위한 다음 작업을 정리합니다.

현재 프로젝트의 핵심 강점은 다음과 같습니다.

- 작업을 설계·구현·검증 단계로 나누고 각 단계에 Sol, Terra, Luna를 배정합니다.
- 부모 모델의 역할을 계산해 불필요한 중복 에이전트를 줄입니다.
- 사용자 승인, 모델 고정 검증, 권한 경계, 실패 폐쇄 원칙을 유지합니다.
- 디자인 작업에서 Sol의 품질 책임과 Terra의 구현 창의성을 함께 보장합니다.

다음 단계에서는 이러한 정책이 실제로 품질, 시간, 토큰 사용량에 어떤 영향을 주는지 측정하고, Codex 버전 변화와 배포 문제를 체계적으로 관리해야 합니다.

---

## 개발 원칙

1. **측정되지 않은 절감률을 주장하지 않습니다.**
2. **Sol 직접 실행을 항상 유효한 기준선으로 유지합니다.**
3. **TOKEN_SAVER에서도 안전·품질 하한을 낮추지 않습니다.**
4. **사용자가 승인하지 않은 모델·권한·작업 범위를 실행하지 않습니다.**
5. **모델 고정 여부를 확인할 수 없으면 성공으로 간주하지 않습니다.**
6. **프롬프트, 소스코드, 비밀정보를 자동 수집하거나 외부로 전송하지 않습니다.**
7. **Codex의 특정 버전에서만 유효한 우회 방법은 날짜와 검증 버전을 함께 기록합니다.**

# P0. 라이선스와 릴리스 정합성 확보

## 문제

MIT `LICENSE` 파일은 현재 기본 브랜치에 존재하지만, 기존 설치 문서가 고정해 둔 이전 릴리스 태그에는 라이선스가 포함되지 않았을 수 있습니다.

저장소의 최신 상태는 오픈소스이더라도, 사용자가 실제로 설치하는 고정 릴리스에 라이선스가 빠져 있으면 다음 문제가 생깁니다.

- 설치된 파일의 이용·수정·재배포 조건이 불명확해집니다.
- 오픈소스 프로그램 신청 시 저장소와 배포물의 조건이 어긋납니다.
- README, 보안 검사, 릴리스 태그가 서로 다른 버전을 가리킬 수 있습니다.

## 작업

- [ ] 다음 패치 릴리스에 `LICENSE`를 포함합니다.
- [ ] 다음 릴리스 버전을 결정합니다. 기본 목표는 `v1.0.3`입니다.
- [ ] `README.md`와 `README.en.md`의 설치 명령을 새 태그로 변경합니다.
- [ ] 보안 검사에 `LICENSE`를 필수 파일로 추가합니다.
- [ ] 보안 검사에서 고정 설치 태그가 README와 일치하는지 확인합니다.
- [ ] 변경 내역에 MIT 라이선스 추가 사실을 명시합니다.
- [ ] 릴리스 태그에서 `LICENSE`, README, 보안 검사 파일이 모두 포함됐는지 확인합니다.
- [ ] 새 릴리스 설치 후 스킬 인식과 보안 검사를 재현합니다.

## 산출물

- `v1.0.3` 릴리스
- 라이선스가 포함된 고정 설치 명령
- 태그 일치 여부를 검사하는 CI 규칙
- 릴리스 검증 체크리스트

## 완료 기준

- 새 환경에서 README의 명령만 사용해 설치할 수 있습니다.
- 설치된 릴리스 루트에 `LICENSE`가 존재합니다.
- `security_check.py`가 새 릴리스에서 통과합니다.
- README, CI, 릴리스 태그가 모두 같은 버전을 참조합니다.

# P1. 비교 벤치마크 구축

## 문제

현재 라우팅 정책은 논리적으로 정교하지만 다음 효과가 실제 수치로 증명되지 않았습니다.

- Sol 직접 실행보다 토큰을 적게 사용하는가?
- 전체 작업 시간이 짧아지는가?
- 결과 품질과 테스트 통과율이 유지되는가?
- 에이전트 분리 비용이 절감 효과보다 커지는 구간은 어디인가?
- PERFORMANCE, BALANCED, TOKEN_SAVER의 차이가 실제로 나타나는가?

따라서 프로젝트를 “좋은 정책 제안”에서 “검증된 라우팅 방법론”으로 발전시키려면 재현 가능한 비교 실험이 필요합니다.

## 벤치마크 구성

최소 20개, 목표 30개 이상의 작업을 준비합니다.

| 작업군 | 예시 | 핵심 평가 항목 |
|---|---|---|
| 소형 수정 | 문구 변경, 설정 한 줄 수정 | 라우팅 오버헤드 여부 |
| 일반 구현 | REST API, 데이터 처리 기능 | 구현 정확성, 테스트 통과율 |
| 대형 구현 | 관리자 대시보드, 다중 기능 앱 | 분업 효과, 통합 품질 |
| 디자인 포함 | 랜딩 페이지, 게임 UI | 주관적 품질, 일관성, 사용성 |
| 대량 변환 | 파일 수백 개 변환, 데이터 정규화 | Luna 활용 가치, 오류율 |
| 코드 조사 | 대형 저장소 구조 분석 | 탐색 속도, 근거 정확성 |
| 검증 작업 | 회귀 테스트, 보안 검토 | 결함 발견률, 오탐률 |
| 고위험 설계 | DB 마이그레이션, 권한 구조 | 판단 품질, 독립 검토 가치 |

각 작업은 가능한 범위에서 다음 네 방식으로 비교합니다.

1. **Sol Direct**
2. **PERFORMANCE**
3. **BALANCED**
4. **TOKEN_SAVER**

## 측정 항목

- [ ] 작업 성공 여부
- [ ] 자동 테스트 통과율
- [ ] 사람이 평가한 결과 품질
- [ ] 재작업 횟수
- [ ] 결함 수와 심각도
- [ ] 실제 경과 시간
- [ ] 생성된 서브에이전트 수
- [ ] 실행된 실제 모델과 추론 강도
- [ ] 토큰 사용량 또는 사용량 관측 가능 여부
- [ ] 라우팅 계획과 실제 실행의 차이
- [ ] 모델 고정 실패 또는 자동 상향 횟수
- [ ] 통합 과정에서 발생한 충돌 수

토큰 사용량을 런타임에서 확인할 수 없으면 숫자를 추정하지 않고 `not observable`로 기록합니다.

## 품질 평가 방식

자동 평가와 사람 평가를 분리합니다.

### 자동 평가

- 테스트 통과율
- 린트와 타입 검사
- 빌드 성공 여부
- 파일 변환 정확도
- 회귀 테스트 결과
- 접근성·대비·오버플로 같은 객관적 UI 검사

### 사람 평가

- 요구사항 충족도
- 코드 구조와 유지보수성
- 설명의 명확성
- 디자인의 일관성·독창성·사용성
- 중요한 위험을 놓치지 않았는지

가능하면 평가자는 어떤 실행 방식의 결과인지 모르는 상태에서 비교합니다.

## 저장소 구조 제안

```text
benchmarks/
├── README.md
├── tasks/
│   ├── small-fix/
│   ├── implementation/
│   ├── design/
│   ├── bulk-transform/
│   └── high-risk/
├── rubrics/
├── results/
│   └── samples/
└── scripts/
```

실제 사용자 프롬프트나 비공개 소스코드는 벤치마크 저장소에 자동 저장하지 않습니다. 공개 가능한 합성 작업이나 별도 허가를 받은 작업만 사용합니다.

## 산출물

- 공개 가능한 표준 작업 세트
- 실행 방법과 평가 기준
- Sol Direct 대비 비교 표
- 프로필별 장단점 보고서
- 라우팅이 이득이 되는 작업과 손해가 되는 작업의 경계

## 완료 기준

- 최소 20개 작업이 네 가지 실행 방식으로 비교됩니다.
- 모든 수치는 재현 방법과 함께 공개됩니다.
- 프로젝트 README에서 근거 없는 절감률 대신 실제 결과를 참조할 수 있습니다.
- 작은 작업, 문맥이 강하게 결합된 작업, 대량 반복 작업의 손익 분기점을 설명할 수 있습니다.

# P1. 실제 행동 테스트 구축

## 문제

현재 보안 검사는 필수 문구와 파일이 존재하는지를 잘 확인하지만, 실제 Codex가 그 정책을 따르는지는 충분히 검증하지 못합니다.

예를 들어 다음 문장이 파일에 있는 것과 실제 행동은 다릅니다.

- 사용자 승인 전에 쓰기 작업을 하지 않는가?
- 부모가 Sol일 때 중복 Sol 계획자를 만들지 않는가?
- Terra를 요청했는데 실제로 Sol이 실행되면 실패로 처리하는가?
- Luna가 애매한 판단을 맡지 않는가?
- 모델 고정이 불가능할 때 몰래 부모 모델을 사용하지 않는가?

## 테스트 계층

### 1. 정적 정책 검사

기존 `security_check.py`를 유지하고 다음을 보강합니다.

- [ ] `LICENSE` 필수 여부
- [ ] README의 설치 태그 일치 여부
- [ ] 오래된 버전·우회 설정에 날짜 표기가 있는지
- [ ] 허용되지 않은 외부 URL 또는 실행 지시가 없는지

### 2. 계획 생성 시나리오 테스트

고정 입력에 대해 라우터가 만들어야 하는 계획의 핵심 조건을 검사합니다.

- [ ] 일반 요청에서는 자동 활성화되지 않음
- [ ] 명시적 호출 후 프로필을 먼저 제시함
- [ ] 프로필 선택 전에는 도구를 사용하지 않음
- [ ] 프로필 선택은 읽기 전용 조사만 허용함
- [ ] 실행 전 별도 승인을 요청함
- [ ] 부모 Sol의 계획 역할을 중복하지 않음
- [ ] 작은 결합 작업은 Sol Direct를 추천함
- [ ] 대량의 객관적 변환만 Luna에 배정함
- [ ] 의미 있는 디자인 작업에는 Sol 책임이 포함됨

### 3. 런타임 통합 테스트

가능한 Codex 환경에서 실제 `spawn_agent` 호출을 검증합니다.

- [ ] 요청한 모델과 실제 자식 모델 일치
- [ ] 요청한 추론 강도와 실제 메타데이터 일치
- [ ] `fork_turns: "none"` 동작 확인
- [ ] 허용된 쓰기 범위 유지
- [ ] 서브에이전트가 다시 하위 에이전트를 만들지 않음
- [ ] 모델 불일치 시 즉시 중단하고 보고
- [ ] 승인 없는 설정 수정이나 재시작을 수행하지 않음

### 4. 실패 주입 테스트

다음 오류를 의도적으로 재현하거나 모의 처리합니다.

- [ ] 모델 필드가 보이지 않음
- [ ] 모델을 지정했으나 부모 모델로 실행됨
- [ ] Multi-Agent V2 설정 충돌
- [ ] 예약된 도구 네임스페이스 충돌
- [ ] CLI와 Desktop 내장 CLI 버전 불일치
- [ ] 테스트 실패로 Luna에서 Terra 또는 Sol로 상향 필요

## 산출물

- 행동 시나리오 테스트 모음
- 실패 주입 테스트 모음
- CI에서 실행되는 정적·행동 검사
- 실제 런타임이 필요한 테스트와 모의 테스트의 분리

## 완료 기준

- 문구가 존재하는지만 확인하는 검사를 넘어 핵심 라우팅 행동을 검증합니다.
- 승인, 모델 고정, 권한 경계, 상향 규칙이 회귀 테스트로 보호됩니다.
- Codex 버전 변화로 동작이 달라지면 CI 또는 호환성 테스트에서 확인할 수 있습니다.

# P1. Codex 호환성 매트릭스 구축

## 문제

이 프로젝트는 Codex의 다음 요소에 직접 의존합니다.

- `spawn_agent` 도구 스키마
- 모델·역할·추론 강도 지정 필드
- Multi-Agent V2 설정
- 도구 네임스페이스
- CLI와 Desktop 앱의 내장 런타임
- 자식 에이전트 메타데이터 표시 방식

이 기능들은 버전에 따라 바뀔 수 있으므로 한 시점의 우회 방법을 영구 규칙으로 취급하면 안 됩니다.

## 작업

- [ ] 공식 CLI 최신 버전과 직전 안정 버전을 구분해 검사합니다.
- [ ] Windows, macOS, WSL에서 최소 검증 범위를 정합니다.
- [ ] Desktop 앱 버전과 내장 CLI 버전을 별도로 기록합니다.
- [ ] PATH의 CLI와 앱 내장 CLI가 다를 때 경고합니다.
- [ ] `spawn_agent`에 노출되는 필드 목록을 자동 기록합니다.
- [ ] 명시적 Terra·Luna 모델 프로브를 실행하는 선택적 검사 도구를 만듭니다.
- [ ] 우회 설정마다 확인 날짜, 대상 버전, 관련 이슈를 기록합니다.
- [ ] 새 버전에서 문제가 해결되면 오래된 우회 설정을 제거합니다.

## 호환성 표 제안

| 환경 | Codex 버전 | 앱 내장 CLI | 모델 지정 | 추론 강도 | 역할 지정 | 검증 결과 |
|---|---|---|---|---|---|---|
| Windows CLI | 기록 | 해당 없음 | 가능/불가 | 가능/불가 | 가능/불가 | A/B/C |
| WSL CLI | 기록 | 해당 없음 | 가능/불가 | 가능/불가 | 가능/불가 | A/B/C |
| macOS CLI | 기록 | 해당 없음 | 가능/불가 | 가능/불가 | 가능/불가 | A/B/C |
| Codex Desktop | 기록 | 기록 | 가능/불가 | 가능/불가 | 가능/불가 | A/B/C |

## 지원 등급

- **A — Full Support:** 모델·역할·추론 강도를 지정하고 실제 실행 메타데이터로 확인할 수 있음
- **B — Limited Support:** 서브에이전트는 실행되지만 모델 고정을 확인할 수 없음
- **C — Request-wide Failure:** 요청이 모델 실행 전에 도구 스키마 오류로 거부됨

B 또는 C에서는 라우팅 성공을 주장하지 않습니다.

## 산출물

- `COMPATIBILITY.md`
- 선택적 런타임 진단 스크립트
- 버전별 알려진 문제 표
- 오래된 우회 방법 제거 기준

## 완료 기준

- 사용자가 자신의 환경이 A, B, C 중 어디에 해당하는지 확인할 수 있습니다.
- 문서가 최신 공식 릴리스 기준으로 갱신됩니다.
- 버전별 조언과 영구 정책이 명확히 구분됩니다.

# P2. 라우팅 정책 보정

## 문제

현재 정책은 안전과 품질을 우선하여 보수적으로 설계되어 있습니다. 이는 장점이지만 일부 작업에서 Sol을 과도하게 사용할 가능성이 있습니다.

특히 “사용자 화면이 존재한다”는 이유만으로 모든 변경에 새로운 Sol 디자인 단계가 필요한 것은 아닙니다.

예를 들어 다음은 서로 다릅니다.

- 새로운 제품 경험과 시각 언어를 설계하는 작업
- 기존 디자인 시스템을 그대로 따르는 단순 폼 추가
- 버튼 문구나 간격만 수정하는 작업
- 내부 직원용 관리 화면의 소규모 유지보수

## 작업

### 디자인 작업 분류 개선

- [ ] `새로운 디자인 판단`과 `기존 디자인 계약 구현`을 구분합니다.
- [ ] 기존 디자인 시스템과 명확한 수용 기준이 있으면 Terra 단독이 가능한 범위를 정의합니다.
- [ ] 주관적 품질이 사업적으로 중요한 경우 Sol 하한을 유지합니다.
- [ ] 사소한 UI 수정에 불필요한 Sol 계획 단계를 추가하지 않도록 예시를 만듭니다.
- [ ] 디자인 방향 변경과 구현 수준 개선의 경계를 명확히 합니다.

### 모델 배정 보정

- [ ] 작업 난이도뿐 아니라 실패 영향도와 검증 가능성을 함께 점수화합니다.
- [ ] Luna 사용은 작업량이 설정·검증 비용을 상쇄할 때만 허용합니다.
- [ ] Terra 구현 후 객관적 검증으로 충분한 작업을 분류합니다.
- [ ] 고위험 작업은 비용 절약보다 독립 검토를 우선합니다.
- [ ] Sol Direct 방어 규칙의 실제 기준을 벤치마크 결과로 보정합니다.

## 판단 축 제안

각 단계에 대해 다음 항목을 낮음·중간·높음으로 평가합니다.

- 모호성
- 실패 영향도
- 작업 반복성
- 객관적 검증 가능성
- 문맥 결합도
- 독립 검토 가치
- 서브에이전트 설정 비용
- 사용자 경험·디자인 중요도

이 점수는 절대 규칙이 아니라 일관된 판단을 돕는 설명 도구로 사용합니다.

## 산출물

- 보정된 `routing-policy.md`
- 작업별 긍정·부정 예시
- 디자인 작업 판정표
- 벤치마크 결과를 반영한 Sol Direct 기준

## 완료 기준

- 단순 UI 유지보수와 새로운 디자인 작업을 구분합니다.
- 모델 배정 이유를 모호성·위험·검증 가능성으로 설명할 수 있습니다.

# P2. 스킬 문맥 비용 최적화

- 라우팅 계획 작성

## 산출물

- 짧은 인계 계약 템플릿
- 문맥 중복 검사 항목
- 최적화 전후 비교 결과

## 완료 기준

# P2. 선택적 로컬 측정 기록

## 문제

실제 사용 데이터를 전혀 축적하지 않으면 라우팅 정책을 경험적으로 개선하기 어렵습니다. 반대로 사용자 프롬프트와 소스코드를 자동 수집하면 개인정보와 보안 문제가 발생합니다.

따라서 **명시적 동의가 있을 때만**, 내용이 아닌 실행 통계만 로컬에 저장하는 방식이 필요합니다.

## 원칙

- 기본값은 기록 안 함입니다.
- 외부 서버로 자동 전송하지 않습니다.
- 프롬프트, 응답 본문, 소스코드, 파일 경로, 비밀정보를 저장하지 않습니다.
- 사용자가 기록을 검토하고 직접 삭제할 수 있어야 합니다.
- 공개 벤치마크에 포함할 때는 별도 동의를 받습니다.

## 기록 형식 예시

```json
{
  "schema_version": 1,
  "task_category": "frontend-build",
  "profile": "balanced",
  "planned_route": ["sol-design", "terra-build", "sol-acceptance"],
  "actual_models": ["gpt-5.6-sol", "gpt-5.6-terra"],
  "elapsed_seconds": 1240,
  "tests_passed": true,
  "rework_count": 1,
  "token_usage": "not observable",
  "model_mismatch": false
}
```

## 작업

- [ ] 통계 스키마를 정의합니다.
- [ ] 저장 금지 필드를 명문화합니다.
- [ ] 로컬 기록 기능은 명시적 선택 사항으로 만듭니다.
- [ ] 기록 미리보기와 삭제 방법을 제공합니다.
- [ ] 스키마 버전을 관리합니다.
- [ ] 공개용 집계 도구는 원본 내용을 읽지 않도록 합니다.

## 산출물

- `metrics.schema.json`
- 개인정보 보호 설명
- 선택적 로컬 기록 예시
- 익명 집계 스크립트

## 완료 기준

- 기본 설정에서는 어떤 측정 파일도 생성하지 않습니다.
- 사용자가 동의한 경우에도 내용 데이터 없이 실행 통계만 저장합니다.
- 저장된 데이터를 사용자가 쉽게 확인하고 삭제할 수 있습니다.

# 성공 지표

프로젝트 성공은 스타 수만으로 평가하지 않습니다.

- 재현 가능한 벤치마크 수
- 실제 사용자 환경에서 확인된 호환성 사례
- 해결된 재현 가능 버그 수
- 프로필별 측정 결과의 신뢰도
- 잘못된 모델 배정률 감소
- Sol Direct 대비 품질을 유지하면서 절약된 실행 사례
- 정책 변경에 대한 테스트 커버리지
- 외부 기여와 이슈 처리 기록

숫자 결과는 측정 조건과 함께 공개하며, 특정 환경에서 얻은 결과를 모든 사용자에게 보장하지 않습니다.
